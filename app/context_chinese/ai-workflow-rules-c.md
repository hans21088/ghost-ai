## 方法

使用規格驅動的工作流程遞增地構建此專案。環境文件定義要構建什麼、如何構建以及目前的進度狀態。始終根據這些規格進行實施 - 不要從頭推斷或發明行為。

## 劃分範圍規則

- 一次只能在一個功能單位或子系統上工作。
- 優先選擇小型、可驗證的增量，而不是大規模推測性的改動。
- 不要在單個實施步驟中組合不相關的系統邊界。

## 何時分割工作

如果實施步驟組合了以下內容，應分割工作：
- UI 改動和背景任務改動
- 即時畫布狀態和資料庫持久化
- 多個不相關的 API 路由
- 環境文件中未清楚定義的行為

如果無法快速進行端到端驗證，則範圍太廣 - 應分割。

## 處理缺失的需求

- 不要發明環境文件中未定義的產品行為。
- 如果需求模糊不清，請在實施前在相關環境文件中解決。
- 如果需求缺失，請在繼續前在 `progress-tracker.md` 中添加為未解決的問題。

## 受保護的基礎元件

除非明確指示，否則不要修改生成的第三方基礎元件。

這包括：
- `components/ui/*`（shadcn/ui 元件）
- 第三方程式庫內部

這些應保持默認和可重用。

專案特定的樣式、佈局改動和功能邏輯必須在應用級別元件中實施，而不是修改基礎元件。

只有在任務明確要求時，才修改這些檔案。

## 保持文件同步

每當實施變更時，更新相關的環境文件：

- 系統架構或邊界
- 儲存模型決策
- 程式碼慣例或標準
- 功能範圍

進度狀態必須反映實施的實際狀態，而非預期狀態。

## 移至下一個單位前

1. 目前單位在其定義的範圍內端到端工作。
2. 未違反 `architecture-context.md` 中定義的任何不變量。
3. `progress-tracker.md` 反映已完成的工作。
