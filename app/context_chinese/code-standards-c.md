# 程式碼標準

## 一般規則

- 保持模組小且單一目的。
- 修復根本原因 - 不要分層解決方法。
- 不要在一個元件或路由中混合不相關的關注點。
- 尊重 'architecture-context.md' 中定義的系統邊界。

## TypeScript

- 整個專案中必須啟用嚴格模式。
- 避免 `any`；使用明確的介面或範圍受限的類型。
- 在信任外部未知輸入前，在系統邊界驗證它。
- 為物件契約使用 `interface`。

## Next.js

- 預設使用 React Server Components。
- 僅當元件需要瀏覽器互動、hook 或即時狀態時，才添加 `"use client"`。
- 保持路由處理程序專注於單一責任。
- 長時間運行的工作屬於背景任務，不屬於請求處理程序。

## 樣式設計

- 使用 `globals.css` 中定義的 CSS 自訂屬性權杖 - 不使用原始 Tailwind 顏色類別（如 `zinc-*`）或硬編碼的十六進位值。
- 透過 Tailwind 工具名稱參考權杖：`bg-base`、`text-copy-primary`、`border-surface-border`、`text-brand` 等。
- 維護邊界半徑規模：小元素使用 `rounded-xl`、卡片使用 `rounded-2xl`、模態窗口使用 `rounded-3xl`。

## API 路由

- 在任何邏輯運行前驗證和解析請求輸入。
- 在任何變更前強制執行驗證和專案所有權檢查。
- 返回一致且可預測的回應形狀。
- 保持路由處理程序簡潔 - 將複雜性推送到共用模組或背景任務。

## 資料和儲存

- 專案中繼資料和關係應透過 Prisma 位於 PostgreSQL 中。
- 畫布快照和生成的規格應位於 Vercel Blob 中；Prisma 僅儲存 blob URL 參考。
- 不要直接在資料庫中儲存大型生成的內容。
- 工作任務運行記錄是一等公民關聯資料 - 在任何權杖簽發前將所有權和運行 ID 視為已驗證。

## 檔案組織

- `lib/` - 共用基礎設施：Prisma 用戶端、驗證協助程式、工具函式。
- `trigger/` - 所有耐久背景任務和 AI 工作流程。
- `components/` - UI 組合唯一；無業務邏輯。
- `app/api/` - 驗證、觸發和持久化的路由處理程序。
- 按文件包含的責任為其命名，而非技術。
